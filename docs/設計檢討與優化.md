# 《房東監視中》設計檢討與優化提案

> 版本:2026-07-10 · 作者:Opus(Fable 週額度已滿,由 Opus 代寫)
> 性質:**只設計、不含實作程式碼**。供之後交給 Sonnet/Opus 逐項實作。
> 讀法:每節皆為「現況 → 問題 → 具體優化(含型別草稿/模組名)→ 優先序」。優先序 = 高(高槓桿、先做)/中/低。

---

## 0. 一句話總評

遊戲的**系統廣度已經很完整**(觀察/經濟/張力/社交/AI 敘事/AI 抉擇/記憶改行為全都有),接下來的高槓桿不在「加更多系統」,而在 **(a) 把 `store.ts` 拆開讓它可維護、(b) 讓數值模型與各系統扣得更緊、(c) 讓 AI 敘事有「跨日的長期劇情弧」而非每天各自為政**。前三名建議:**A. store 模組化 → B. 數值模型重整(含啟用 hygiene/新增 energy)→ C. 劇情弧與記憶生命週期**。

---

## 1. AI 生成文本

### 現況
- `worker/index.ts` 的 `/api/narrate` 一次呼叫 Gemini,回 `{diary, newMemory, event}`。
- context 由 `store.buildNarrateCtx()` 組:當天日誌、性格、記憶、關係、鄰居、房東抉擇。
- 每「遊戲日」換日、且為 live 時觸發(每租客一次);離線/超額 fallback 模板(`narrate.ts` + `content/observationLines.ts`)。

### 問題
1. **每天各自為政**:AI 只看「當天」片段 + 靜態記憶,缺「昨天寫了什麼」的連續感 → 容易同質、劇情不連貫。
2. **context 沒有壓縮/摘要**:租客住越久,memoryTags/關係越多,prompt 越長、成本越高、重點越糊。
3. **額度脆弱**:Gemini 免費層每分鐘/每日上限低,4 租客同時換日 = 瞬間 4 次;快轉跨多日會爆。
4. **無快取/去重**:相似情境重複呼叫。

### 優化設計
- **加「連續性摘要」**:每位租客維護一段滾動 `narrativeSummary`(≤200 字),每次 AI 回傳時**順便更新它**(prompt 要求輸出 `summaryUpdate`),下次把它餵進去。→ 用一個短摘要取代「塞一堆歷史」,又保連貫。
  ```ts
  interface NarrateResult { diary; newMemory; event; summaryUpdate?: string }  // worker 回傳新增
  TenantRuntime.narrativeSummary: string                                      // 取代/補強 tenant.recentSummary
  ```
- **context 預算化**:`buildNarrateCtx` 只帶「最近 N 筆重點日誌 + 最新摘要 + 最多 K 個最相關記憶 + 進行中關係」,其餘丟摘要。
- **佇列/節流**:換日時把多位租客的 narrate 請求**排隊**(間隔 ≥ RPM 限制),而非並發;可在 `store` 加一個簡單的 `narrationQueue`。
- **供應商抽象**:worker 內把「呼叫哪個模型」抽成 `providers/{gemini,claude}.ts`,方便日後換 2.0/2.5 或加重試/退避(429 自動退避 N 秒)。
- **可選:輕量快取**:對「狀態雷同、無新記憶」的日子直接走模板,省呼叫。
- 優先序:連續性摘要 **高**;節流/退避 **高**(額度痛點);context 預算化 **中**;快取 **低**。

## 2. AI 影響劇情(讓 AI 更深、更安全地影響世界)

### 現況
- AI 事件可影響單一租客,或**牽涉第二位鄰居**(`other`/`rel`,嚴格夾值、取向把關);記憶標籤已能**改行為**(`memoryEffects.ts`)。

### 問題
1. **沒有「長期劇情弧」**:事件是一次性的,缺「連載」——第 3 天埋的伏筆第 10 天回收。
2. **記憶會無限累積又同權**:8 個上限、先進先出,沒有「淡出/強化/演化」(失戀該慢慢好起來)。
3. **事件彼此獨立**:缺因果連鎖(A 事件的選擇 → 解鎖 B 事件)。

### 優化設計
- **記憶生命週期**:給記憶加 `intensity`(0~1)與 `decayPerDay`,每日衰減;`memoryDrift` 乘上 intensity;衰到 0 自動移除。讓「失戀」隨時間淡化、痊癒。
  ```ts
  interface MemoryTag { …; intensity: number; decayPerDay: number }
  ```
- **劇情弧 (Arc)**:每位租客可有 0~1 條進行中的 arc(`ThreadState`),AI 每日推進一步;arc 提供「連載」骨架。
  ```ts
  interface StoryArc { id; theme: string; stage: number; maxStage: number; summary: string }
  TenantRuntime.arc?: StoryArc
  ```
  prompt 告知目前 arc 與 stage,要求 AI「推進或收束」。
- **事件連鎖**:AI 事件的選擇可留一個 `flag`(記入租客),之後 `rollEvent`/AI 依 flag 解鎖後續事件。
- **關係網 → 群體事件**:目前社交是兩兩;可加「三角關係/室友公約/公共衝突」等牽涉 3+ 人的事件(需擴充事件 target 為多人)。
- **安全**:所有新增影響一律延續現有原則(store 端夾值、玩家抉擇、不開放驅逐、名字要對得上 roster、忽略未知欄位)。arc/flag 也要 schema 驗證。
- 優先序:記憶生命週期 **高**(便宜、直接讓劇情更真);劇情弧 **中**;事件連鎖 **中**;群體事件 **低**。

## 3. UI / UX

### 現況
- 樓層總覽(可捲動)+ 房間細看;面板:商店/招租/收支/關係/抉擇/同居/家具資訊。
- 提示靠樓層徽章(未讀數、紅點事件)+ toast。

### 問題
1. **動線分散**:功能都塞在樓層底部一排按鈕,面板各自為政;缺一個「今天發生了什麼」的彙整入口。
2. **看故事 vs 做經營混在一起**:玩家想「讀劇情」時要一間間點進房。
3. **事件提示弱**:紅點容易忽略;抉擇是遊戲的高潮卻不夠醒目。
4. **房間細看美術債**:仍是舊 3/4,與樓層 tile 比例不一致(已知)。

### 優化設計
- **首頁加「動態消息 (Feed) 分頁」**:把所有租客的重要日誌/事件/關係變化匯成一條時間軸(像社群動態),玩家一眼看完全棟今天的故事;點某則 → 跳對應房間。這是「看故事」的主動線。
- **事件中心**:所有待決抉擇集中一個醒目入口(數字 badge + 顏色),而非散在各房紅點。
- **底部導覽列**:把「樓層 / 動態 / 收支 / 關係」做成固定 4 分頁,取代一排浮動按鈕,手機更好按。
- **房間細看重畫**:改用與樓層一致的 tile 比例(重用 `floorScene` 的繪法放大),並補齊看電視/用餐/看書等狀態 sprite。
- **日夜光線**:樓層依 `gameMs` 疊一層時段色調,增加「時間在走」的臨場感。
- 優先序:動態 Feed **高**(直接放大核心體驗);底部導覽 **中**;事件中心 **中**;房間細看重畫 **中**(美術工);日夜 **低**。

## 4. 租客數值模型(重點)

### 現況
- `Tenant.stats = { mood, stress, hygiene, affinity }` + `TenantRuntime.satisfaction`(衍生)。
- **問題最集中的地方**:
  1. **`hygiene` 幾乎沒用**(只有記憶漂移剛接上,無日常來源/後果)。
  2. **會飽和**:mood/stress 容易黏在 0 或 100。
  3. **語意重疊**:mood 與 satisfaction、stress 與 satisfaction 部分重疊。
  4. **和經濟脫節**:租客沒有「自己的錢/財務壓力」,繳租只看 reliability+affinity+satisfaction。

### 優化設計(建議的數值軸重整)
把數值分成**三類**,語意清楚、彼此有因果:

**A. 即時情緒(短期、每小時波動)**
- `mood` 心情、`stress` 壓力、`energy`(**新增**,精力:睡覺/放鬆↑、工作/加班↓;energy 低 → 效率差、易 stress)。

**B. 狀態(中期、緩慢變化)**
- `hygiene` → 改名/擴為 `wellbeing`(身心健康:運動/衛生↑、熬夜/高壓↓),並**給它後果**(低 → 生病事件、mood 上限下降)。
- `affinity` 對房東好感(保留)。

**C. 衍生(唯讀,由上面算)**
- `satisfaction` 居住滿意(現有公式再納入 wellbeing/energy)。
- **新增 `socialFulfillment`**:由社交關係算出(有朋友/戀人↑、孤立↓)→ 影響 mood,讓社交系統回饋到數值。

**租客財務(新增,和經濟扣緊)**
```ts
Tenant.wallet: { balance: number; monthlyIncome: number }   // 租客自己的收支
```
- 租客有收入/開銷;`monthsOverdue` 真正由 wallet 決定 → 「繳不出租」變成有根據的劇情,而非隨機。

**抗飽和機制**
- 所有每小時 delta 改成**朝基準值回歸 (homeostasis)**:`stat += (baseline - stat) * k + event/memory/routine deltas`。個性決定 baseline(工作狂 stress baseline 高)。→ 自然不黏 0/100,且性格差異體現在「回到哪」。

- 優先序:抗飽和(homeostasis)**高**(治本、影響全域手感);啟用/重定義 wellbeing + 後果 **高**;energy **中**;socialFulfillment **中**;租客 wallet **中**。
- 風險:改數值軸會牽動 satisfaction/事件/社交/收租多處,需集中一次重構 + 大量回歸測試(`sim-trace`、新增 balance 快照測試)。

## 5. 整體架構 / 擴充性 / 效能 / 資料模型

### 現況與問題
1. **`store.ts` 過大**(~700+ 行,tick/作息/事件/社交/AI/收支/存檔/擺放全在一起)→ 難維護、易改壞。
2. **存檔**:單一 localStorage blob、手寫欄位、`?? 預設` 式相容;版本欄位 `v:2` 但無正式遷移層。
3. **reactive 效能**:每 rAF 重繪整個 canvas;租客多時 `Object.values(runtimes)` 反覆掃描。目前規模 OK,但要留意。
4. **內容資料驅動**:家具/文本已資料化;事件(規則式)、作息仍寫死在程式。

### 優化設計
- **拆 store 成多個 slice/模組**(仍共用一個 reactive `state`):
  - `sim/tick.ts`(hourlyTick + applyHour + 偏離)、`economy.ts`(addMoney/collectRent/upkeep/ledger)、`tenancy.ts`(moveIn/moveOut/招租/佔用/配色)、`narration.ts`(每日 AI/摘要/佇列)、`persistence.ts`(save/load/migrate)。`store.ts` 只留 `state` 定義 + 對外 re-export。
- **正式存檔遷移層**:`persistence.ts` 內 `migrate(raw): SaveV3`,一段一段版本升級;`SAVE_VERSION` 常數;壞檔安全 fallback。
- **事件與作息資料化**:把 `events.ts` 的規則式事件、`routine.ts` 的作息抽成 `data/*.json` 或 catalog 式陣列,讓「加事件/改作息 = 改資料」。
- **效能**(有需要再做):agents/canvas 只在「有變化」時重繪;`occupancy`/`runtimes` 掃描結果快取。
- **測試策略**:除了 `sim-trace`,加「**balance 快照測試**」——固定隨機種子跑 N 天,快照關鍵數值/金錢曲線,改動時 diff,避免手感悄悄跑掉。
- 優先序:store 拆分 **高**(所有後續工作的地基);存檔遷移層 **高**(改資料結構前必備);事件/作息資料化 **中**;效能 **低**;balance 快照測試 **中**。

## 6. 建議實作順序(可交給 Sonnet/Opus 的獨立工作項)

> 每項:目標 · 涉及檔案 · 相依 · 難度 · 優先序

1. **store 模組化重構** · 拆成 tick/economy/tenancy/narration/persistence · 全 store 呼叫點 · 相依:無(先做)· 難度 中 · **高**
2. **存檔遷移層 + SAVE_VERSION** · `persistence.ts` · 相依:1 · 難度 低-中 · **高**
3. **數值模型重整(homeostasis + 啟用 wellbeing + energy)** · `types.ts`/`generate.ts`/`store(tick)`/`updateSatisfaction` · 相依:1 · 難度 高 · **高**
4. **AI 連續性摘要 + 節流/退避** · `worker`/`narration.ts`/`buildNarrateCtx` · 相依:1 · 難度 中 · **高**
5. **記憶生命週期(intensity/decay)** · `types.ts`/`memoryEffects.ts`/`narration` · 相依:3 佳 · 難度 低-中 · **中**
6. **首頁動態 Feed + 底部導覽** · `App.vue`/新元件 · 相依:無 · 難度 中 · **中**
7. **劇情弧 (StoryArc) + 事件連鎖 flag** · `types.ts`/`events`/`worker`/`narration` · 相依:4,5 · 難度 中-高 · **中**
8. **事件/作息資料化** · `data/*`、`events.ts`/`routine.ts` · 相依:1 · 難度 中 · **中**
9. **房間細看重畫(tile 一致)+ 新狀態 sprite + 日夜** · `scene.ts`/`PixelDollhouse`/`floorScene` · 相依:無 · 難度 中(美術)· **中**
10. **balance 快照測試** · `scripts/` · 相依:3 · 難度 低 · **中**

**最推薦的前三**:**1 → 2 → 3**(地基:先讓程式可維護、存檔可遷移,再動數值模型)。之後 **4(AI 連續性)** 與 **6(動態 Feed)** 對「玩起來的感覺」提升最明顯。

## 附:不建議現在做 / 風險提醒
- **多人共享世界**:需帳號 + DB + 同步,是另一個量級的工程,建議等單機體驗打磨完再評估(見工作日誌待辦)。
- **一次改太多數值軸**:務必集中在單一 PR + 快照測試護航,否則手感會悄悄崩。
- **AI 影響力放太寬**:任何讓 AI 動到世界狀態的新能力,都要維持「夾值 + 玩家抉擇 + schema 驗證」三道關卡。
